# Modulo Projetos - Implementation Plan

> **For agentic workers:** implementar task por task, com commits pequenos. Como ha tabela nova, RLS, storage e a segunda excecao de visibilidade do cliente no app, este plano deve passar por revisao previa do Claude Code em modo somente leitura antes da implementacao.

**Commit-base:** `61145e7`

**Responsavel pela implementacao futura:** Codex.

**Revisor previo:** Claude Code, somente leitura.

**Spec de referencia:** `docs/superpowers/specs/2026-07-18-projetos-design.md`

**Goal:** Criar o modulo Projetos, um repositorio de documentos da obra (projeto executivo,
memorial, administrativo) com versionamento simples — nova revisao vira automaticamente a
atual, historico preservado — visivel em modo leitura para o cliente e gerenciavel por
admin/equipe com o modulo `projetos`.

**Architecture:** O modulo nasce com duas tabelas (`projetos_documentos`,
`projetos_revisoes`), um enum de categoria, RLS por obra e permissao por modulo `projetos`,
mais um bucket de Storage privado so-PDF (`projetos`) com policy de leitura estendida ao
papel `cliente` — diferente da maioria dos buckets do app, que hoje excluem cliente. A UI usa
uma pagina de lista + uma pagina de detalhe em React, rotas protegidas seguindo o padrao dos
demais modulos, com a rota liberada tambem para `cliente` (leitura).

**Tech Stack:** Supabase/Postgres + RLS + Storage; React 19 + TypeScript + Vite 6; CSS
modules existentes.

## Global Constraints

- Documento pertence obrigatoriamente a uma obra; sem vinculo com unidade/etapa/servico.
- Todo documento nasce com pelo menos uma revisao — nao existe documento vazio.
- So PDF e aceito no upload (`application/pdf`).
- Codigo de revisao (`revisao`) e texto livre digitado pelo usuario — nao gerado automaticamente.
- Nova revisao vira `atual = true` automaticamente; a revisao anterior do mesmo documento vira
  `atual = false` na mesma transacao.
- Cliente enxerga a lista e baixa qualquer revisao (atual ou historica) — leitura apenas, sem
  botoes de escrita.
- Escrita (criar documento, subir revisao, editar, inativar) restrita a admin e equipe com o
  modulo `projetos`.
- Isolamento por obra via `pode_acessar_obra(obra_id)` com policy `AS RESTRICTIVE`, criada
  na MESMA migracao inicial — nunca como retrofit posterior (ver achado critico da revisao
  previa de Tarefas, `docs/superpowers/plans/2026-07-18-tarefas.md`).
- Bucket de Storage precisa ser adicionado a policy RESTRICTIVE existente
  `isolamento_obra_storage` (`supabase/migrations/20260717_isolamento_usuario_obra.sql`) —
  sem isso, os arquivos do bucket `projetos` ficam SEM isolamento entre obras, porque essa
  policy so aplica a restricao aos buckets explicitamente listados nela.
- Soft delete segue a regra do projeto: SELECT precisa permitir ver linha inativa para quem
  pode editar `projetos`, evitando bloqueio de `ativo = false`.
- Toda resposta de erro do Supabase deve ser tratada na tela.
- Sem `window.confirm`; usar dialogo existente do app (`useConfirmDialog`).
- Enum novo em `modulo_app` precisa ser aplicado em transacao separada do resto da migracao
  (Postgres nao permite usar um valor de enum recem-criado na mesma transacao — ver nota no
  Step 8 do Task 1, aprendida na aplicacao de Tarefas em 18/07/2026).
- Funcoes de trigger `SECURITY DEFINER` precisam de `REVOKE ALL ... FROM PUBLIC, anon,
  authenticated` para nao ficarem expostas como RPC publico (ver achado da auditoria de
  17/07/2026 e do hardening de Tarefas em 18/07/2026).

- [ ] **Step 4: Criar tabela `projetos_revisoes`**

```sql
CREATE TABLE projetos_revisoes (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  documento_id  UUID NOT NULL REFERENCES projetos_documentos(id) ON DELETE CASCADE,
  revisao       TEXT NOT NULL,
  path          TEXT NOT NULL,
  observacao    TEXT,
  atual         BOOLEAN NOT NULL DEFAULT true,
  criado_em     TIMESTAMPTZ NOT NULL DEFAULT now(),
  criado_por    UUID NOT NULL DEFAULT auth.uid() REFERENCES perfis_usuario(id),
  CONSTRAINT projetos_revisoes_revisao_not_blank CHECK (btrim(revisao) <> '')
);

CREATE INDEX idx_projetos_revisoes_documento ON projetos_revisoes(documento_id, criado_em DESC);
CREATE UNIQUE INDEX idx_projetos_revisoes_unica_atual ON projetos_revisoes(documento_id) WHERE atual;
```

O indice unico `idx_projetos_revisoes_unica_atual` garante no banco que nunca existam duas
revisoes `atual = true` do mesmo documento ao mesmo tempo, mesmo que o trigger do Step 5 tenha
algum bug — mesmo padrao de defesa em profundidade usado em
`idx_medicoes_itens_unico_por_medicao` (`supabase/migrations/20260713_fase7_medicoes_travas.sql`).

- [ ] **Step 5: Trigger para marcar a nova revisao como atual**

```sql
CREATE OR REPLACE FUNCTION marcar_revisao_atual()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  UPDATE projetos_revisoes
  SET atual = false
  WHERE documento_id = NEW.documento_id
    AND atual = true;
  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_marcar_revisao_atual
  BEFORE INSERT ON projetos_revisoes
  FOR EACH ROW EXECUTE FUNCTION marcar_revisao_atual();
```

O trigger roda `BEFORE INSERT`: desmarca a revisao anterior do mesmo documento ANTES da nova
linha ser efetivamente inserida. Isso e necessario porque o indice unico parcial do Step 4
(`idx_projetos_revisoes_unica_atual`) e verificado no momento em que cada linha e inserida —
indices unicos parciais nao podem ser `DEFERRABLE` no Postgres. Se o trigger fosse `AFTER
INSERT`, a nova linha (que ja nasce com `atual = true`) coexistiria brevemente com a revisao
antiga (tambem `atual = true`) no momento da insercao, e o proprio `INSERT` falharia com
`duplicate key value violates unique constraint` antes do trigger ter chance de rodar — ou
seja, toda segunda revisao de um documento quebraria. Como a nova linha ainda nao existe na
tabela no momento do `BEFORE INSERT`, nao e preciso excluir `NEW.id` da clausula `WHERE`.

- [ ] **Step 7: Policies de `projetos_documentos` e `projetos_revisoes`**

```sql
-- Isolamento por obra desde o dia 1 (nao como retrofit).
CREATE POLICY isolamento_obra ON projetos_documentos AS RESTRICTIVE FOR ALL TO authenticated
  USING (pode_acessar_obra(obra_id))
  WITH CHECK (pode_acessar_obra(obra_id));

CREATE POLICY isolamento_obra ON projetos_revisoes AS RESTRICTIVE FOR ALL TO authenticated
  USING (EXISTS (SELECT 1 FROM projetos_documentos d WHERE d.id = documento_id AND pode_acessar_obra(d.obra_id)))
  WITH CHECK (EXISTS (SELECT 1 FROM projetos_documentos d WHERE d.id = documento_id AND pode_acessar_obra(d.obra_id)));

-- Leitura: todos os papeis, inclusive cliente (segunda excecao do app, apos Definicoes de Projeto).
-- Escrita: admin, ou equipe com o modulo 'projetos'.
CREATE POLICY projetos_documentos_select ON projetos_documentos FOR SELECT TO authenticated
  USING (ativo = true OR pode_editar_projetos());
CREATE POLICY projetos_documentos_insert ON projetos_documentos FOR INSERT TO authenticated
  WITH CHECK (pode_editar_projetos());
CREATE POLICY projetos_documentos_update ON projetos_documentos FOR UPDATE TO authenticated
  USING (pode_editar_projetos())
  WITH CHECK (pode_editar_projetos());

CREATE POLICY projetos_revisoes_select ON projetos_revisoes FOR SELECT TO authenticated
  USING (EXISTS (
    SELECT 1 FROM projetos_documentos d
    WHERE d.id = documento_id AND (d.ativo = true OR pode_editar_projetos())
  ));
CREATE POLICY projetos_revisoes_insert ON projetos_revisoes FOR INSERT TO authenticated
  WITH CHECK (pode_editar_projetos() AND EXISTS (
    SELECT 1 FROM projetos_documentos d WHERE d.id = documento_id AND d.ativo = true
  ));
```

Nao ha policy de UPDATE/DELETE em `projetos_revisoes`: revisao e append-only no MVP (corrigir
= subir nova revisao), igual ao padrao ja usado em `tarefas_comentarios`.

Note que a policy de SELECT de `projetos_documentos` NAO restringe por papel (nem exclui
`cliente`) — isso e intencional, e a forma como o cliente ve o modulo em modo leitura, igual
ao padrao ja usado em `definicoes_projeto`.

- [ ] **Step 8: Criar o bucket de Storage e suas policies**

```sql
INSERT INTO storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
VALUES ('projetos', 'projetos', false, 20971520, ARRAY['application/pdf'])
ON CONFLICT (id) DO NOTHING;

-- Leitura: todos os papeis, inclusive cliente. Escrita: admin/equipe com o modulo.
CREATE POLICY projetos_storage_select ON storage.objects FOR SELECT TO authenticated
  USING (bucket_id = 'projetos');
CREATE POLICY projetos_storage_insert ON storage.objects FOR INSERT TO authenticated
  WITH CHECK (bucket_id = 'projetos' AND pode_editar_projetos());
```

**Passo critico — nao pular:** a policy RESTRICTIVE `isolamento_obra_storage`, criada em
`supabase/migrations/20260717_isolamento_usuario_obra.sql`, so aplica isolamento entre obras
aos buckets explicitamente listados nela (`rdo`, `fvs`, `pendencias`, `cotacoes-nf`). Um
bucket novo que NAO for adicionado a essa lista fica **sem isolamento entre obras** — qualquer
usuario autenticado, de qualquer obra, conseguiria ler/escrever arquivos de qualquer obra
nesse bucket, apesar da policy `projetos_storage_select`/`projetos_storage_insert` acima (essas
so checam papel, nunca obra). E preciso recriar essa policy adicionando `projetos`:

```sql
DROP POLICY isolamento_obra_storage ON storage.objects;

CREATE POLICY isolamento_obra_storage ON storage.objects AS RESTRICTIVE FOR ALL TO authenticated
USING (
  bucket_id NOT IN ('rdo','fvs','pendencias','cotacoes-nf','projetos')
  OR CASE
    WHEN bucket_id IN ('rdo','fvs','pendencias','projetos') THEN
      split_part(name,'/',1) ~* '^[0-9a-f]{8}-[0-9a-f]{4}-[1-5][0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$'
      AND pode_acessar_obra(split_part(name,'/',1)::UUID)
    WHEN bucket_id = 'cotacoes-nf' THEN
      split_part(name,'/',1) ~* '^[0-9a-f]{8}-[0-9a-f]{4}-[1-5][0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$'
      AND EXISTS (
        SELECT 1 FROM pedidos_compra p
        WHERE p.id=split_part(name,'/',1)::UUID AND pode_acessar_obra(p.obra_id)
      )
    ELSE false
  END
)
WITH CHECK (
  bucket_id NOT IN ('rdo','fvs','pendencias','cotacoes-nf','projetos')
  OR CASE
    WHEN bucket_id IN ('rdo','fvs','pendencias','projetos') THEN
      split_part(name,'/',1) ~* '^[0-9a-f]{8}-[0-9a-f]{4}-[1-5][0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$'
      AND pode_acessar_obra(split_part(name,'/',1)::UUID)
    WHEN bucket_id = 'cotacoes-nf' THEN
      split_part(name,'/',1) ~* '^[0-9a-f]{8}-[0-9a-f]{4}-[1-5][0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$'
      AND EXISTS (
        SELECT 1 FROM pedidos_compra p
        WHERE p.id=split_part(name,'/',1)::UUID AND pode_acessar_obra(p.obra_id)
      )
    ELSE false
  END
);
```

Isso exige que todo `path` gravado em `projetos_revisoes.path` comece com `obra_id/` (UUID)
como primeira pasta — mesmo padrao usado em RDO/FVS/Pendencias (ex.:
`{obra_id}/{documento_id}/{revisao}.pdf`).

- titulo preenchido;
- categoria selecionada;
- arquivo selecionado e do tipo `application/pdf` (checar `file.type === 'application/pdf'`,
  com mensagem clara se nao for);
- codigo de revisao preenchido;
- obra ativa existente.

- [ ] **Step 5: Criar documento + primeira revisao**

Fluxo ao salvar:

1. `INSERT` em `projetos_documentos` (retornando o `id` criado);
2. `upload` do arquivo no bucket `projetos`, path `${obraAtiva.id}/${documentoId}/${Date.now()}.pdf`

3. `INSERT` em `projetos_revisoes` com o `path` retornado do upload.

Se o passo 2 ou 3 falhar depois do passo 1 ja ter criado o documento, exibir erro claro e
deixar o documento (sem revisao) visivel na lista — o usuario pode entrar no detalhe e subir a
revisao de novo (Task 3 cobre "nova revisao" para documento existente, reaproveitavel aqui). Nao

### Critico 1 — trigger `AFTER INSERT` em conflito com o indice unico parcial: toda segunda revisao falharia

- **Modulo/cenario:** Projetos — subir a segunda revisao (ou qualquer revisao alem da
  primeira) de um documento existente.
- **Arquivo/trecho:** Task 1, Step 4 (`idx_projetos_revisoes_unica_atual`) combinado com
  Step 5 (`trg_marcar_revisao_atual ... AFTER INSERT ON projetos_revisoes`).
- **Evidencia:** o indice unico parcial `ON projetos_revisoes(documento_id) WHERE atual` e
  verificado no momento em que CADA LINHA e inserida, nao no fim da transacao (indices
  unicos parciais nao podem ser `DEFERRABLE` no Postgres — deferrable exige constraint de
  tabela inteira, sem `WHERE`). Como a nova revisao nasce com `atual = true` (default da
  coluna) e a revisao anterior do mesmo documento AINDA esta com `atual = true` no momento do
  `INSERT`, a verificacao do indice unico roda ANTES do trigger `AFTER INSERT` ter chance de
  desmarcar a revisao antiga — a insercao falha com `duplicate key value violates unique
  constraint`. O trigger `AFTER INSERT` nunca chega a executar, porque o proprio `INSERT` ja
  falhou. Conferido: nao ha nenhum precedente desse padrao (flag exclusiva + trigger + indice
  unico parcial) em nenhuma migracao existente do projeto — os padroes de "so um X por vez"
  existentes (ex.: `idx_medicoes_itens_unico_por_medicao`) protegem contra duplicidade de uma
  FK, nao contra a propria linha que esta sendo inserida.
- **Impacto:** o recurso central do modulo (nova revisao) quebraria em todo documento que ja
  tivesse pelo menos uma revisao — ou seja, a partir do segundo upload. Isso passaria batido em
  qualquer teste manual que so criasse documentos novos com uma revisao cada, e so apareceria
  ao testar exatamente o caso de uso que da nome ao modulo (versionamento).
- **Correcao recomendada:** trocar o trigger de `AFTER INSERT` para `BEFORE INSERT`, e
  simplificar a funcao — como a nova linha ainda nao existe na tabela no momento do `BEFORE
  INSERT`, nao e preciso excluir `NEW.id` da clausula `WHERE`:

  ```sql
  CREATE OR REPLACE FUNCTION marcar_revisao_atual()
  RETURNS TRIGGER
  LANGUAGE plpgsql
  SECURITY DEFINER
  SET search_path = public
  AS $$
  BEGIN
    UPDATE projetos_revisoes
    SET atual = false
    WHERE documento_id = NEW.documento_id
      AND atual = true;
    RETURN NEW;
  END;
  $$;

  CREATE TRIGGER trg_marcar_revisao_atual
    BEFORE INSERT ON projetos_revisoes
    FOR EACH ROW EXECUTE FUNCTION marcar_revisao_atual();
  ```

  Isso garante que a revisao antiga e desmarcada ANTES da nova ser efetivamente inserida,
  entao o indice unico nunca ve duas linhas `atual = true` ao mesmo tempo. O `SECURITY
  DEFINER` continua necessario, pelo mesmo motivo ja documentado no plano (nao ha policy de
  UPDATE em `projetos_revisoes` para usuarios comuns).
- **Teste de validacao:** criar um documento com uma revisao, subir uma segunda revisao
  (Task 3 Step 2) e confirmar que funciona sem erro de constraint; conferir que so a revisao
  mais nova fica com `atual = true` e a anterior aparece no historico como superada.
- **Status:** defeito comprovado de desenho — reproduzivel deterministicamente, nao um risco
  teorico.

### Medio 2 — codigo de revisao (texto livre) usado sem saneamento no path do storage

- **Modulo/cenario:** Projetos — nome do arquivo no bucket ao subir documento/revisao.
- **Arquivo/trecho:** Task 2 Step 5 e Task 3 Step 2 — path proposto
  `${obraAtiva.id}/${documentoId}/${Date.now()}-${revisao}.pdf`, onde `revisao` e o texto
  livre digitado pelo usuario (Global Constraints e spec §3.1 confirmam que e livre, sem
  validacao de formato).
- **Impacto:** um usuario pode digitar algo como "R00/final" ou "Rev. 01 (aprovada)" como
  codigo de revisao — caracteres como `/` quebram a estrutura de pastas esperada pelo path
  (criaria uma subpasta extra), e outros caracteres especiais podem gerar nomes de arquivo
  confusos ou, dependendo do cliente Supabase, rejeitados pelo Storage. Nao chega a ser uma
  falha de isolamento (o primeiro segmento do path, `obra_id`, continua controlado pelo
  frontend), mas pode gerar arquivos com path inconsistente.
- **Correcao recomendada:** sanear `revisao` antes de usar no path (ex.: substituir
  qualquer caractere que nao seja letra/numero/hifen por `_`), mantendo o texto original
  intacto na coluna `projetos_revisoes.revisao` (que e so exibida na tela, nunca usada como
  path). Alternativa mais simples: no path, usar o `id` da propria revisao (gerado pelo
  banco) em vez do texto digitado — ex. `${obraAtiva.id}/${documentoId}/${revisaoId}.pdf` —
  eliminando o problema por completo sem precisar de saneamento.
- **Teste de validacao:** criar uma revisao com codigo `"R00/teste"` ou `"Rev. 01 (v2)"` e
  confirmar que o arquivo e salvo com um path unico e previsivel, sem criar subpastas
  inesperadas.
- **Status:** defeito comprovado de desenho (o path realmente ficaria inconsistente com esse
  input), severidade media por depender de um input especifico do usuario, nao do fluxo
  padrao.

### Baixo 3 — sem `atualizado_por`/`atualizado_em` ao editar um documento

  trigger `BEFORE UPDATE`, ou aceitar a mesma lacuna ja existente em Definicoes de Projeto
  por consistencia. Nao bloqueante.
- **Status:** sugestao, nao defeito — decisao de escopo, ja tem precedente aceito no
  projeto.

### Baixo 4 — policy de leitura do storage nao verifica `ativo` do documento pai

- **Modulo/cenario:** Projetos — acessar o arquivo de uma revisao apos o documento ser
  inativado.
- **Evidencia:** `projetos_storage_select` (Task 1 Step 8) so checa `bucket_id = 'projetos'`
  (mais o isolamento por obra da policy RESTRICTIVE); nao verifica se
  `projetos_documentos.ativo = true`. Um cliente que ja tivesse acesso a uma signed URL
  gerada antes da inativacao (validade 300s) ou que retivesse o `path` de alguma forma
  poderia gerar uma nova signed URL para um documento ja inativado.
- **Impacto:** baixo — signed URLs expiram em 5 minutos, e o cenario exige acesso previo
  legitimo ao documento. Alem disso, esse e o mesmo padrao ja aceito em todos os outros
  buckets do projeto (RDO, FVS, Pendencias): a policy de storage nunca verifica o `ativo` da
  tabela pai, so a policy da TABELA controla isso. Nao e uma regressao, e uma limitacao
  preexistente e consistente do modelo de storage do app.
- **Correcao recomendada:** nenhuma agora — mencionado para registro; se algum dia o projeto
  decidir fechar essa lacuna, deveria ser feito de forma centralizada (todos os buckets),
  nao so em Projetos.
- **Status:** observacao, nao bloqueante.

## Bug encontrado no teste de campo do Rodrigo — 18/07/2026 (debugging sistematico)

**Sintoma:** ao testar como admin, criar documento funcionou mas o upload do PDF falhou com
"new row violates row-level security policy".

**Investigacao (Fase 1 — causa raiz, evidencia real, nao suposicao):**

- Testei a policy `projetos_storage_insert` e o bucket `projetos` via SQL direto — ambos
  corretos, identicos ao que foi revisado.
- Consultei o documento recem-criado: `obra_id = '00000000-0000-0000-0000-000000000001'`,
  `criado_por` = Rodrigo (admin, perfil ativo).
- Testei o regex da policy `isolamento_obra_storage` contra esse `obra_id` exato via SQL:

**Causa raiz confirmada:** o `obra_id` da obra piloto e um UUID seedado manualmente na Fase 0

(`supabase/migrations/20260717_isolamento_usuario_obra.sql`) exige explicitamente uma versao
1-5 e variante RFC4122 — mais restritivo que "isso e um UUID valido". Esse regex ja existia
antes desta sessao e cobre os buckets `rdo`, `fvs`, `pendencias` e `cotacoes-nf`. A migracao de
Projetos so *copiou* esse padrao (ja quebrado) para o bucket novo.

**Escopo real do bug:** nao e um problema exclusivo de Projetos. Qualquer upload novo para os
buckets `rdo`, `fvs`, `pendencias` ou `cotacoes-nf` na obra piloto falharia com o mesmo erro,
desde que a policy `isolamento_obra_storage` foi criada em 17/07/2026. Verifiquei que nao ha
nenhuma foto de RDO com data de 18/07/2026 no banco (a mais recente e de 17/07) — nao deu pra
confirmar com o Rodrigo se RDO/FVS/Pendencias ja tinham sido tentados hoje.

**Correcao aplicada (Fase 4):** troquei o regex por um validador de formato UUID generico
(8-4-4-4-12 em hexadecimal, sem exigir versao/variante especificas), via nova migracao
`supabase/migrations/20260718_fix_regex_uuid_storage.sql`, aplicada em producao. Confirmado via
SQL que o regex novo aceita o `obra_id` da obra piloto; `get_advisors` nao apontou nenhum
achado novo.

**Recomendacao:** ao testar Projetos de novo, tambem vale confirmar que anexar foto em um RDO
novo funciona — se ja estava quebrado desde 17/07, esta correcao resolve os dois casos ao
mesmo tempo.
