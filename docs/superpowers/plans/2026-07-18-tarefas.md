# Modulo Tarefas - Implementation Plan

> **For agentic workers:** implementar task por task, com commits pequenos. Como ha tabela nova, RLS e regras de permissao, este plano deve passar por revisao previa do Claude Code em modo somente leitura antes da implementacao.

**Commit-base:** `c07769f`

**Responsavel pela implementacao futura:** Codex.

**Revisor previo:** Claude Code, somente leitura.

**Spec de referencia:** `docs/superpowers/specs/2026-07-18-tarefas-design.md`

**Goal:** Criar o modulo Tarefas para registrar providencias operacionais da obra/escritorio, com responsavel, prazo obrigatorio, prioridade, status, comentarios/historico, vinculo opcional com EAP e contador de atrasadas no Dashboard.

**Architecture:** O modulo nasce com duas tabelas (`tarefas`, `tarefas_comentarios`), enums proprios, RLS por obra e permissao por modulo `tarefas`. A UI usa uma pagina dedicada em React, rotas protegidas pelo mesmo padrao dos modulos existentes, e Dashboard recebe um contador simples de tarefas atrasadas e abertas do usuario. Fotos/anexos ficam fora do MVP.

**Tech Stack:** Supabase/Postgres + RLS; React 19 + TypeScript + Vite 6; CSS modules existentes.

## Global Constraints

- Prazo e obrigatorio no formulario e no banco.
- Toda tarefa pertence obrigatoriamente a uma obra.
- Unidade, etapa e servico sao opcionais.
- Equipe so pode concluir tarefa em que esteja como responsavel.
- Admin pode concluir qualquer tarefa.
- Reabrir tarefa concluida/cancelada e exclusivo do admin.
- Cliente nao ve o modulo e nao acessa a rota/tabelas.
- Comentarios sao append-only no MVP.
- Fotos/anexos ficam para depois do MVP.
- Soft delete deve seguir a regra do projeto: SELECT precisa permitir ver linha inativa para quem pode editar, evitando bloqueio de `ativo = false`.
- Toda resposta de erro do Supabase deve ser tratada na tela.
- Sem `window.confirm`; usar dialogo existente do app.

- [ ] **Step 1: Conferir enum `modulo_app` existente**

Verificar se `tarefas` ja existe no enum `modulo_app`. Se nao existir, adicionar via bloco seguro:

```sql
DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1
    FROM pg_enum e
    JOIN pg_type t ON t.oid = e.enumtypid
    WHERE t.typname = 'modulo_app'
      AND e.enumlabel = 'tarefas'
  ) THEN
    ALTER TYPE modulo_app ADD VALUE 'tarefas';
  END IF;
END $$;
```

- [ ] **Step 2: Criar enums**

```sql
CREATE TYPE status_tarefa AS ENUM ('aberta', 'em_andamento', 'concluida', 'cancelada');
CREATE TYPE prioridade_tarefa AS ENUM ('baixa', 'normal', 'alta', 'urgente');
```

Se o projeto ja tiver padrao de `DO $$` para enums idempotentes, seguir o padrao local.

- [ ] **Step 3: Criar tabelas**

Criar `tarefas` com:

- `obra_id NOT NULL`;
- `prazo DATE NOT NULL`;
- campos opcionais `unidade_id`, `etapa_id`, `servico_id`;
- status/prioridade com defaults;
- `responsavel_id`;
- campos de conclusao/cancelamento;
- `ativo`, `criado_por`, `criado_em`, `atualizado_em`.

Criar `tarefas_comentarios` com:

- `tarefa_id`;
- `tipo` (`comentario` ou evento textual controlado pelo app);
- `comentario`;
- `criado_por`, `criado_em`.

- [ ] **Step 4: Constraints de integridade**

Adicionar constraints:

- `titulo` nao vazio;
- `motivo_cancelamento` obrigatorio quando status for `cancelada`;
- `concluida_por/concluida_em` obrigatorios quando status for `concluida`;
- `cancelada_por/cancelada_em` obrigatorios quando status for `cancelada`;
- `unidade_id`, `etapa_id`, `servico_id` precisam pertencer a mesma obra quando preenchidos, se houver helper/padrao local simples para isso.

- [ ] **Step 5: Funcoes e RLS**

Criar `pode_editar_tarefas()` seguindo o padrao de `pode_editar_*` existente:

- `admin` retorna true;
- `equipe` com modulo `tarefas` retorna true;
- cliente retorna false.

RLS:

- SELECT em `tarefas`: usuario autenticado com acesso a obra e (`ativo = true OR pode_editar_tarefas()`), cliente excluido.
- INSERT em `tarefas`: `pode_editar_tarefas()` e obra acessivel.
- UPDATE em `tarefas`: `pode_editar_tarefas()` e obra acessivel, com regras de status:
  - equipe nao conclui tarefa de outro responsavel;
  - reabrir concluida/cancelada so admin;
  - cliente nunca atualiza.
- SELECT em `tarefas_comentarios`: permitido para quem pode ler a tarefa.
- INSERT em `tarefas_comentarios`: permitido para quem pode editar tarefas e ler a tarefa.
- UPDATE/DELETE em comentarios: nao permitir no MVP.

- [ ] **Step 6: Tipos TypeScript**

Adicionar em `src/lib/supabase.ts`:

- `StatusTarefa`;
- `PrioridadeTarefa`;
- `Tarefa`;
- `TarefaComentario`.
